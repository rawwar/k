# Aider — Core Architecture

## High-Level Design

Aider is a Python application (~30k lines) structured around a central `Coder` class hierarchy that mediates between the user, LLMs, the file system, and git. The architecture is designed around a key insight: **different LLMs need different strategies for expressing code edits**, so the edit format is a pluggable, first-class abstraction.

```
aider/
├── main.py                  # CLI entry point, argument parsing
├── coders/                  # Core: the Coder class hierarchy
│   ├── base_coder.py        # Base class — chat loop, context assembly
│   ├── editblock_coder.py   # "diff" edit format (search/replace blocks)
│   ├── wholefile_coder.py   # "whole" edit format (full file replacement)
│   ├── udiff_coder.py       # "udiff" edit format (unified diff)
│   ├── architect_coder.py   # Architect mode (two-model orchestration)
│   ├── editor_editblock_coder.py  # Editor sub-coder for architect mode
│   ├── editor_whole_coder.py      # Editor sub-coder (whole format)
│   ├── *_prompts.py         # System prompts for each edit format
│   └── __init__.py          # Registry of all coder types
├── repomap.py               # Tree-sitter repo map generation
├── repo.py                  # Git repository abstraction
├── commands.py              # Slash-command handling (/add, /drop, /undo, etc.)
├── io.py                    # Terminal I/O, pretty printing, input handling
├── models.py                # Model configuration, routing, capabilities
├── sendchat.py              # LLM API interaction (via litellm)
├── voice.py                 # Voice-to-text via Whisper
├── watch.py                 # File watcher for IDE integration
├── linter.py                # Built-in and custom linting
├── prompts.py               # Shared prompt templates
├── queries/                 # Tree-sitter query files (.scm) per language
└── website/                 # Docs site (aider.chat)
```

## The Coder Class Hierarchy

The heart of Aider is the `Coder` base class in `base_coder.py`. This is a large (~2000 line) class that handles:

- **Chat history management** — maintaining message lists for the LLM
- **Context assembly** — combining repo-map, added files, and user messages into prompts
- **Response parsing** — extracting file edits from LLM responses
- **File application** — writing edits to disk
- **Git integration** — auto-committing changes
- **Lint/test cycle** — running linters and tests after edits

Each **edit format** is implemented as a subclass:

```
Coder (base_coder.py)
├── EditBlockCoder        # "diff" format — search/replace blocks
├── WholeFileCoder        # "whole" format — full file output
├── UnifiedDiffCoder      # "udiff" format — unified diffs
├── ArchitectCoder        # "architect" mode — two-model pipeline
├── EditBlockFuncCoder    # "diff-func" — function-call variant
└── WholeFileFuncCoder    # "whole-func" — function-call variant
```

The `Coder.create()` factory method selects the right subclass based on the model's configured edit format:

```python
@classmethod
def create(cls, main_model, edit_format, io, **kwargs):
    # Looks up the appropriate Coder subclass for the edit_format
    # e.g., "diff" → EditBlockCoder, "whole" → WholeFileCoder
```

## Model Routing (`models.py`)

Aider uses **litellm** as its LLM abstraction layer, allowing it to connect to virtually any provider. The `models.py` module handles:

1. **Model identification** — parsing model names (e.g., `sonnet`, `o3-mini`, `deepseek/deepseek-chat`)
2. **Capability detection** — determining if a model supports vision, function calling, streaming, thinking tokens
3. **Edit format selection** — choosing the optimal edit format per model:
   - Claude models → `diff`
   - Gemini models → `diff-fenced` (Gemini struggles with standard fence formatting)
   - GPT-3.5 → `whole` (too weak for diff formats)
   - o1/o3 reasoning models → `architect` mode recommended
   - DeepSeek → `diff`
4. **Weak model selection** — choosing a cheaper model for tasks like commit messages
5. **Token limit management** — knowing context windows and pricing per model

Model metadata is configured through a combination of:
- Built-in defaults in `models.py`
- A `model-metadata.json` file
- User overrides via `--model-metadata-file`

## Git Integration (`repo.py`)

The `GitRepo` class wraps git operations:

- **Auto-commit**: Every edit is committed with a descriptive message generated by the "weak model"
- **Dirty file protection**: Uncommitted changes are committed separately before AI edits begin
- **Undo**: `/undo` reverts the last AI commit
- **Diff tracking**: Shows what changed after each interaction
- **Attribution**: Commits by aider are marked with `(aider)` in the author name

## Commands System (`commands.py`)

Aider provides ~30 slash commands:

| Command | Purpose |
|---------|---------|
| `/add <file>` | Add file to chat context |
| `/drop <file>` | Remove file from context |
| `/ask <question>` | Ask without editing |
| `/code <request>` | Request code changes |
| `/architect <request>` | Use architect mode |
| `/undo` | Revert last AI change |
| `/diff` | Show recent changes |
| `/run <cmd>` | Run shell command, optionally share output |
| `/test <cmd>` | Run tests, auto-fix failures |
| `/lint` | Run linter on changed files |
| `/voice` | Voice input via Whisper |
| `/model <name>` | Switch LLM |
| `/tokens` | Show token usage |
| `/clear` | Clear chat history |
| `/map` | Show current repo map |
| `/map-refresh` | Force repo map refresh |

## Terminal I/O (`io.py`)

Aider uses `prompt_toolkit` for rich terminal interaction:
- Syntax-highlighted code display
- Markdown rendering of LLM responses
- Tab completion for commands and file paths
- Streaming output with live diffs
- Color-coded display (user input, AI output, tool output, errors)

## Watch Mode (`watch.py`)

The `--watch-files` mode enables IDE integration:
- Monitors all repo files for changes
- Scans for special `AI` / `AI!` / `AI?` comments
- Triggers code edits or questions automatically
- Allows users to work in their preferred editor while aider runs in a terminal

## Configuration System

Aider supports configuration through multiple layers (in priority order):
1. Command-line arguments
2. `.aider.conf.yml` in the repo root
3. `.aider.conf.yml` in the home directory
4. Environment variables (`AIDER_xxx`)
5. `.env` file
