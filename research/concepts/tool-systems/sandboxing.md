# Sandboxing

How coding agents isolate tool execution to prevent dangerous side effects — from Linux kernel primitives to cloud sandbox services.

---

## 1. Why Sandbox?

Code execution is inherently dangerous. Every `subprocess.run()`, every `exec()`, every
shell invocation is a doorway to the full power of the operating system. For traditional
software, this is manageable — developers write the code, review it, test it. But coding
agents introduce a fundamentally different threat model: **the code being executed is
generated by an LLM**, and LLMs can be wrong, confused, or manipulated.

### The Risks Are Real

**File system destruction.** An agent asked to "clean up the project" might `rm -rf /`
if the model hallucinates. More subtly, it might delete `.git/` or overwrite
configuration files, causing damage that is hard to detect and harder to reverse.

**Data exfiltration.** A prompt injection hidden in a README could instruct the agent
to `curl` environment variables (containing API keys, tokens) to an attacker's server.
This is not hypothetical — researchers have demonstrated this attack against multiple
agent frameworks.

**Cryptocurrency mining.** An agent with unrestricted compute access could be tricked
into downloading and running a miner. The user pays for the compute; the attacker profits.

**Supply chain attacks.** An agent running `pip install` or `npm install` could pull in
a malicious package that executes arbitrary code during installation. The agent itself
becomes the attack vector into the developer's environment.

**Lateral movement.** On a developer's machine, an agent has access to SSH keys,
cloud credentials, browser cookies, and other secrets. A compromised agent could
pivot to production systems.

### The Fundamental Tension

Agents NEED to modify files and run commands — that is their entire purpose. A coding
agent that cannot write code or run tests is useless. But unrestricted access to the
host system is dangerous. Sandboxing is the art of finding the narrow path between
"useful enough to do real work" and "restricted enough to prevent catastrophe."

The challenge is that the boundary between "intended side effect" and "dangerous side
effect" is context-dependent. Writing to `src/main.py` is expected; writing to
`/etc/passwd` is not. Running `pytest` is fine; running `curl` to an unknown server
is suspicious. Sandboxing must encode these distinctions.

```
  +---------------------------------------------+
  |  LLM generates code/commands                |
  |  (may be wrong, malicious, or manipulated)  |
  +---------------------------------------------+
  |           SANDBOX BOUNDARY                  |
  |  +---------------------------------------+  |
  |  |  Controlled execution environment     |  |
  |  |  * Limited filesystem access          |  |
  |  |  * Restricted network                 |  |
  |  |  * Filtered syscalls                  |  |
  |  |  * Resource limits                    |  |
  |  +---------------------------------------+  |
  +---------------------------------------------+
  |  Host system (protected)                    |
  +---------------------------------------------+
```

## 2. Codex's 3-Layer Approach — The Most Sophisticated

OpenAI's Codex CLI implements the most layered sandbox architecture among open-source
coding agents, combining three independent Linux security mechanisms that compose
defensively — if one layer fails, the others still protect the system.

### Layer 1: bubblewrap (User Namespace Isolation)

[bubblewrap](https://github.com/containers/bubblewrap) (`bwrap`) is a lightweight
unprivileged sandboxing tool that uses Linux namespaces to create isolated environments
without requiring root access.

Codex uses bwrap to create five types of isolation:

**Mount namespace.** The entire filesystem is bind-mounted read-only by default. Only
explicitly specified directories (the project workspace, `/tmp`) are mounted writable.
The agent can read system libraries and tools but cannot modify them.

**PID namespace.** Processes inside the sandbox cannot see or signal processes outside.
The sandboxed process tree is completely isolated — `kill -9 1` inside the sandbox
kills only the sandbox's init process, not the host's.

**User namespace.** UID/GID mapping creates an unprivileged user inside the sandbox
even if the outer process has elevated privileges. This prevents privilege escalation.

**Network namespace.** All networking is blocked by default. The sandbox has no
network interfaces except loopback. Crucially, only `AF_UNIX` (Unix domain) sockets
are permitted — this allows the agent to communicate with its parent process via
a pre-established socket without allowing arbitrary network access.

**IPC namespace.** Shared memory segments and semaphores are isolated, preventing
inter-process communication with host processes.

Example bwrap invocation (simplified from Codex source):

```bash
bwrap \
  --ro-bind / /                          \
  --bind /home/user/project /workspace   \
  --bind /tmp /tmp                       \
  --tmpfs /run                           \
  --dev /dev                             \
  --proc /proc                           \
  --unshare-pid                          \
  --unshare-net                          \
  --unshare-ipc                          \
  --unshare-user                         \
  --uid 1000 --gid 1000                  \
  --die-with-parent                      \
  --new-session                          \
  -- /bin/bash -c "$COMMAND"
```

The flags break down as:
- `--ro-bind / /` — Mount the entire root filesystem read-only
- `--bind ... /workspace` — Mount the project directory writable
- `--unshare-pid` — Create new PID namespace
- `--unshare-net` — Create new network namespace (no connectivity)
- `--unshare-user` — Create new user namespace
- `--die-with-parent` — Kill sandbox if parent process dies (prevents orphans)

### Layer 2: seccomp (Syscall Filtering)

Even within an isolated namespace, the kernel surface is large. seccomp-bpf (Secure
Computing mode with Berkeley Packet Filters) allows a process to install a filter
that restricts which system calls it can make.

Codex installs a seccomp filter that:

**Blocks dangerous syscalls:**
- `ptrace` — prevents debugging/tracing other processes
- `io_uring_setup`, `io_uring_enter`, `io_uring_register` — blocks io_uring which
  has been a source of kernel exploits
- `socket(AF_INET, ...)`, `socket(AF_INET6, ...)` — blocks TCP/UDP network sockets
- `mount`, `umount2` — prevents filesystem manipulation
- `reboot`, `kexec_load` — prevents system disruption
- `init_module`, `finit_module` — prevents kernel module loading

**Allows only AF_UNIX sockets** for IPC with the parent process:

```c
// Pseudocode for seccomp BPF filter
struct sock_filter filter[] = {
    // Load syscall number
    BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
             offsetof(struct seccomp_data, nr)),

    // Block ptrace
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_ptrace, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ERRNO | EPERM),

    // For socket(), inspect the domain argument
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_socket, 0, 4),
    BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
             offsetof(struct seccomp_data, args[0])),
    // Allow AF_UNIX (domain = 1)
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, AF_UNIX, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),
    // Block all other socket domains
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ERRNO | EPERM),

    // Default: allow
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),
};
```

The key insight is filtering at the *argument* level, not just the syscall level.
The `socket()` syscall itself is allowed, but only with `AF_UNIX` as the domain
argument. This enables inter-process communication while blocking network access.

### Layer 3: Landlock (Filesystem Access Control)

[Landlock](https://landlock.io/) is a Linux Security Module (available since kernel
5.13) that enables unprivileged processes to restrict their own filesystem access.
Unlike traditional LSMs (SELinux, AppArmor) which require admin configuration,
Landlock rules are self-imposed by the process.

Codex uses Landlock as a defense-in-depth layer and as a fallback when bubblewrap
is not available (e.g., in containers where user namespaces are disabled):

```c
// Pseudocode for Landlock ruleset
struct landlock_ruleset_attr attr = {
    .handled_access_fs =
        LANDLOCK_ACCESS_FS_READ_FILE |
        LANDLOCK_ACCESS_FS_WRITE_FILE |
        LANDLOCK_ACCESS_FS_EXECUTE |
        LANDLOCK_ACCESS_FS_MAKE_DIR |
        LANDLOCK_ACCESS_FS_REMOVE_FILE,
};

int ruleset_fd = landlock_create_ruleset(&attr, sizeof(attr), 0);

// Allow read-only access to /usr, /lib, /etc
add_path_rule(ruleset_fd, "/usr",
    LANDLOCK_ACCESS_FS_READ_FILE | EXECUTE);
add_path_rule(ruleset_fd, "/lib",
    LANDLOCK_ACCESS_FS_READ_FILE | EXECUTE);
add_path_rule(ruleset_fd, "/etc",
    LANDLOCK_ACCESS_FS_READ_FILE);

// Allow full access to workspace
add_path_rule(ruleset_fd, "/workspace", ALL_ACCESS);

// Enforce! (irreversible for this process)
landlock_restrict_self(ruleset_fd, 0);
```

The critical property of Landlock is that it is **irreversible** — once a process
calls `landlock_restrict_self()`, it cannot remove or weaken the restrictions. Even
if the agent gains code execution, it cannot escape the Landlock sandbox.

### How the Three Layers Compose

```
  Process Start
    |
    v
  +--------------------------------------+
  |  Landlock: restrict filesystem paths |  <-- Self-imposed path restrictions
  |  (read-only: /usr, /lib, /etc)       |
  |  (read-write: /workspace, /tmp)      |
  +--------------------------------------+
                 |
                 v
  +--------------------------------------+
  |  bubblewrap: create namespaces       |  <-- OS-level process isolation
  |  * Mount namespace (ro root)         |
  |  * PID namespace                     |
  |  * Network namespace (no net)        |
  |  * User namespace (unprivileged)     |
  +--------------------------------------+
                 |
                 v
  +--------------------------------------+
  |  seccomp: filter syscalls            |  <-- Kernel-level syscall filtering
  |  * Block: ptrace, io_uring, mount    |
  |  * Allow: AF_UNIX sockets only       |
  +--------------------------------------+
                 |
                 v
  +--------------------------------------+
  |  Execute command                     |
  +--------------------------------------+
```

Each layer defends against different failure modes:
- Landlock fails? bubblewrap's read-only mounts prevent writes.
- bubblewrap fails? seccomp blocks dangerous syscalls.
- seccomp bypassed? Landlock still restricts filesystem access.
- All three must be defeated simultaneously for a complete escape.

### Cross-Platform Support

**macOS: Seatbelt** (`sandbox-exec`). Codex uses Apple's `sandbox-exec` with a
custom Scheme-based profile that denies network access and restricts filesystem
writes. Seatbelt is less granular than the Linux stack but provides meaningful
isolation on macOS:

```scheme
(version 1)
(deny default)
(allow process-exec)
(allow file-read*)
(allow file-write*
  (subpath "/Users/user/project")
  (subpath "/tmp"))
(deny network*)
```

**Windows:** Restricted tokens, ACLs, and Windows Firewall rules provide isolation,
though with less granularity than the Linux namespace approach.

### Proxy Routing for Controlled Network Access

When agents need network access (e.g., to install packages), Codex uses a
TCP-to-UDS-to-TCP bridge. The agent connects to a Unix domain socket (allowed by
seccomp), which is bridged by the host process to actual TCP, applying allowlists:

```
  Agent (sandboxed)          Host Process              Internet
  +----------+     UDS      +------------+    TCP    +----------+
  | curl pkg | ---------->  | Proxy with | -------> | Registry |
  |          |   (allowed)  | allowlist  | (filtered)|         |
  +----------+              +------------+           +----------+
```

Only connections to approved domains (PyPI, npm registry, etc.) are forwarded.
All other connection attempts are rejected by the proxy. This gives agents the
ability to install dependencies while preventing data exfiltration.

## 3. Docker-Based: OpenHands

[OpenHands](https://github.com/All-Hands-AI/OpenHands) (formerly OpenDevin) takes a
different approach: every action executes inside a Docker container. This provides
strong isolation through the container boundary but at higher startup cost.

### Architecture

```
  +------------------------------------------------------------+
  |  Host System                                               |
  |                                                            |
  |  +------------------+      HTTP        +----------------+  |
  |  |  AgentController |  <------------>  |  Docker        |  |
  |  |                  |                  |  Container     |  |
  |  |  * LLM interface |                  |  +----------+  |  |
  |  |  * Event stream  |                  |  | Action   |  |  |
  |  |  * State machine |                  |  | Execution|  |  |
  |  |                  |                  |  | Server   |  |  |
  |  +------------------+                  |  +----------+  |  |
  |                                        |  +----------+  |  |
  |                                        |  | Jupyter  |  |  |
  |                                        |  | Kernel   |  |  |
  |                                        |  +----------+  |  |
  |                                        |  +----------+  |  |
  |                                        |  | Browser  |  |  |
  |                                        |  |(Playwrght)|  |  |
  |                                        |  +----------+  |  |
  |                                        |  +----------+  |  |
  |                                        |  | Bash     |  |  |
  |                                        |  | Shell    |  |  |
  |                                        |  +----------+  |  |
  |                                        +----------------+  |
  +------------------------------------------------------------+
```

### How It Works

1. **EventStreamRuntime** creates a Docker container from a pre-built image containing
   all necessary tools (Python, Node.js, git, etc.).

2. **ActionExecutionServer** runs inside the container, listening on a port mapped to
   the host. It receives `Action` objects (shell commands, file writes, browser
   actions) over HTTP and returns `Observation` results.

3. **All tools execute inside the container:**
   - Shell commands via `bash`
   - Code execution via an IPython/Jupyter kernel
   - Web browsing via Playwright (headless Chromium inside the container)
   - File operations scoped to the container filesystem

4. **The project directory is volume-mounted** into the container, giving the agent
   read-write access to the code being worked on while isolating everything else.

### Trade-offs

**Advantages:**
- Battle-tested isolation (Docker has years of security hardening)
- Full Linux environment inside the container
- Easy to reproduce — containers are deterministic
- Network can be controlled via Docker network policies
- Clean environment for each task (no state leakage)

**Disadvantages:**
- Container startup adds 2-10 seconds of latency
- Docker daemon must be running (not always available in CI/restricted envs)
- Volume mounts can be tricky (permission issues, symlinks)
- Heavier resource usage than namespace-only approaches
- Image size can be large (1-5 GB for full development environments)

OpenHands also supports a **remote sandbox runtime** where containers run on a
separate machine, providing even stronger isolation at the cost of network latency.

## 4. gVisor

[gVisor](https://gvisor.dev/) is Google's application kernel — a fundamentally
different approach to sandboxing that does not fit neatly into the "namespace" or "VM"
categories.

### What gVisor Is (and Is Not)

gVisor is NOT a syscall filter like seccomp. Seccomp decides whether to allow or
block a syscall; gVisor **re-implements** the syscall entirely. When a sandboxed
process calls `open()`, gVisor's Sentry component handles the call in userspace
Go code, never passing it to the host kernel.

gVisor is NOT a virtual machine. There is no hardware virtualization, no guest kernel
running in a VM. It is a userspace process that pretends to be a kernel.

```
  Traditional Container          gVisor Container
  +--------------+               +--------------+
  |  Application |               |  Application |
  +------+-------+               +------+-------+
         | syscall                       | syscall
         v                               v
  +--------------+               +--------------+
  |  Host Kernel |               |  Sentry      |  <-- Userspace Go process
  |  (shared!)   |               |  (gVisor)    |      re-implements syscalls
  +--------------+               +------+-------+
                                        | limited syscalls
                                        v
                                 +--------------+
                                 |  Host Kernel |  <-- Much smaller surface
                                 +--------------+
```

### Key Properties

- **OCI-compatible** via the `runsc` runtime — drop-in replacement for `runc`
- **Works with Docker and Kubernetes**: `docker run --runtime=runsc`
- **~5-15% CPU overhead** for compute-bound workloads; higher for syscall-heavy loads
- **~350 of ~400 Linux syscalls** implemented
- **Used in production** by Google Cloud Run, GKE Sandbox

### When to Use for Agents

gVisor is ideal when you need container-like ease of use with VM-like security. For
agent sandboxing, it provides stronger isolation than Docker's default `runc` runtime
because even if the agent finds a container escape exploit, it would escape into
gVisor's userspace kernel, not the real host kernel.

```bash
# Run agent container with gVisor
docker run --runtime=runsc \
  -v /path/to/project:/workspace \
  --network=none \
  agent-image:latest
```

The overhead is acceptable for coding agents because the bottleneck is typically
LLM API latency (seconds), not syscall overhead (microseconds).

gVisor also provides **Gofer**, a separate process that handles filesystem operations.
This means even the Sentry (which handles all other syscalls) cannot directly access
the host filesystem — true defense in depth.

## 5. E2B

[E2B](https://e2b.dev/) provides cloud sandboxes purpose-built for AI code execution.
Rather than building sandbox infrastructure yourself, E2B offers sandboxes as a
service via API.

### How It Works

Each sandbox is a lightweight microVM (based on Firecracker) that boots in ~150ms.
The sandbox has its own filesystem, network stack, and process tree — fully isolated
from other sandboxes and the host.

```python
from e2b import Sandbox

# Create a sandbox (boots in ~150ms)
sandbox = Sandbox()

# Execute code
result = sandbox.run_code("print('Hello from sandbox!')")
print(result.text)  # "Hello from sandbox!"

# Run shell commands
proc = sandbox.commands.run("ls -la /")

# Upload/download files
sandbox.files.write("/home/user/data.csv", csv_content)
content = sandbox.files.read("/home/user/output.txt")

# Sandbox auto-destroys after timeout (default 5 min)
sandbox.close()
```

### Templates

E2B supports custom templates that define the sandbox environment:

```dockerfile
# e2b.Dockerfile
FROM e2b/base:latest
RUN pip install numpy pandas matplotlib
RUN npm install -g typescript
COPY config/ /home/user/.config/
```

```bash
e2b template build -d e2b.Dockerfile
# Returns a template ID to use when creating sandboxes
```

### Key Features

- **Python and JavaScript SDKs** for easy integration
- **Self-hostable on GCP** for data sovereignty requirements
- **Persistent sandboxes** — can pause and resume
- **Internet access** — sandboxes can reach the internet (configurable)
- **File system snapshots** — save and restore sandbox state
- **Streaming output** — real-time stdout/stderr

### Integration with Agent Frameworks

E2B integrates with LangChain, CrewAI, and other frameworks as a tool:

```python
from e2b import Sandbox
from langchain.tools import tool

@tool
def execute_code(code: str) -> str:
    """Execute Python code in a sandboxed environment."""
    sandbox = Sandbox(template="python-data-science")
    result = sandbox.run_code(code)
    sandbox.close()
    return result.text
```

## 6. Daytona

[Daytona](https://github.com/daytonaio/daytona) is an open-source (AGPL-3.0) sandbox
infrastructure designed specifically for coding agents and AI code execution.

### Key Differentiators

**Sub-90ms creation time.** Daytona sandboxes spin up faster than Docker containers,
using optimized microVM technology and pre-warmed pools.

**Built-in LSP support.** Unlike most sandbox solutions that only offer shell access,
Daytona provides Language Server Protocol integration, enabling agents to get code
intelligence (autocomplete, diagnostics, go-to-definition) inside the sandbox.

**Unlimited persistence.** Sandboxes can persist indefinitely, unlike E2B's
timeout-based model. This suits long-running agent tasks.

**Multi-language SDKs.** Python, TypeScript, and Go SDKs are available:

```python
from daytona_sdk import Daytona

daytona = Daytona()
sandbox = daytona.create()

# Execute commands
response = sandbox.process.exec("python3 -c 'print(1+1)'")
print(response.output)  # "2"

# File operations
sandbox.fs.write_file("/home/user/main.py", "print('hello')")
content = sandbox.fs.read_file("/home/user/main.py")

# LSP integration
diagnostics = sandbox.lsp.get_diagnostics("/home/user/main.py")

sandbox.delete()
```

### Architecture

Daytona runs sandboxes as microVMs managed by a control plane. The control plane
handles scheduling, networking, and lifecycle management. Each sandbox gets its
own network namespace with configurable internet access.

## 7. Modal

[Modal](https://modal.com/) is a serverless compute platform that can serve as
sandbox infrastructure for coding agents.

### How It Works

Modal runs functions in isolated containers with automatic scaling. For agent
sandboxing, you define a function that executes in a fresh container:

```python
import modal

app = modal.App("agent-sandbox")
image = modal.Image.debian_slim().pip_install("numpy", "pandas")

@app.function(image=image, timeout=300)
def execute_in_sandbox(code: str) -> str:
    import subprocess
    result = subprocess.run(
        ["python3", "-c", code],
        capture_output=True, text=True, timeout=60
    )
    return result.stdout + result.stderr

# From agent code:
with app.run():
    output = execute_in_sandbox.remote("print('hello')")
```

### Key Features for Agent Sandboxing

- **GPU support** — agents can run ML code with GPU acceleration
- **Function-level isolation** — each invocation gets a fresh container
- **Persistent volumes** — shared storage across invocations if needed
- **Network control** — configurable egress rules
- **Sub-second cold starts** for pre-built images
- **Built-in secrets management** — pass credentials without exposing to agent

## 8. Firejail

[Firejail](https://firejail.wordpress.com/) is a lightweight Linux sandboxing tool
that uses namespaces and seccomp-bpf, similar to bubblewrap but with a different
philosophy.

### Firejail vs bubblewrap

| Aspect          | Firejail                      | bubblewrap                  |
|-----------------|-------------------------------|-----------------------------|
| Philosophy      | Feature-rich, desktop-focused | Minimal, composable         |
| Profiles        | 1000+ app-specific profiles   | No built-in profiles        |
| Setuid          | Runs as setuid root           | Unprivileged (user ns)      |
| Network         | Built-in network management   | Namespace only              |
| X11             | X11 sandboxing support        | No GUI support              |
| Attack surface  | Larger (more features)        | Smaller (minimal code)      |
| Best for        | Desktop app isolation         | Programmatic embedding      |

### Usage for Agents

```bash
# Run agent command in Firejail sandbox
firejail --noprofile \
  --net=none \
  --private=/tmp/sandbox \
  --read-only=/usr \
  --seccomp \
  --noroot \
  --caps.drop=all \
  -- python3 agent_script.py
```

Firejail's security profiles make it easy to sandbox specific tools:

```ini
# /etc/firejail/python3.profile
include /etc/firejail/disable-common.inc
net none
no3d
nosound
seccomp
caps.drop all
noroot
private-tmp
```

For agent use cases, bubblewrap is generally preferred because it is designed
to be embedded programmatically, while Firejail is designed for interactive
desktop use. However, Firejail's pre-built profiles can be useful for quickly
sandboxing specific tools without writing custom configuration.

## 9. Gemini CLI's Multi-Tier Approach

Google's Gemini CLI implements a tiered sandboxing strategy that assigns different
isolation levels to different tool types, recognizing that not all operations carry
equal risk.

### Platform-Specific Sandboxing

**macOS: Seatbelt profiles.** Gemini CLI uses Apple's `sandbox-exec` with custom
profiles for shell command execution:

```scheme
(version 1)
(deny default)
(allow process-fork)
(allow process-exec)
(allow file-read* (subpath "/usr") (subpath "/Library"))
(allow file-read* (subpath "/path/to/project"))
(allow file-write* (subpath "/path/to/project"))
(deny network*)
(allow sysctl-read)
```

**Linux: Docker containers.** When Docker is available, shell commands execute
inside lightweight containers. When Docker is unavailable, Gemini falls back to
reduced-privilege execution.

### Per-Tool Risk Classification

Gemini CLI classifies tools by risk level:

```
  High Risk (always sandboxed):
  |-- Shell command execution
  |-- Code execution
  +-- Package installation

  Medium Risk (configurable):
  |-- File write operations
  |-- File delete operations
  +-- Git operations

  Low Risk (no sandbox):
  |-- File read operations
  |-- Directory listing
  +-- Search/grep
```

This tiered approach reduces overhead — read operations run at native speed while
dangerous operations pay the sandbox cost. The configuration is adjustable, allowing
users to tighten or loosen restrictions based on their trust level.

### Key Design Decision

Unlike Codex (which sandboxes everything uniformly), Gemini CLI's approach
acknowledges that the cost-benefit of sandboxing varies by operation. Reading a
file is inherently safe; there is no security benefit to sandboxing it. This
selective approach reduces latency for common operations.

## 10. mini-SWE-agent: No Sandbox (Natural Isolation)

[mini-SWE-agent](https://github.com/princeton-nlp/SWE-agent) demonstrates that
minimal architectures can achieve reasonable isolation through statelessness rather
than explicit sandboxing.

### How It Works

Each action is a fresh `subprocess.run()` call:

```python
def execute_action(command: str, timeout: int = 30) -> str:
    result = subprocess.run(
        ["bash", "-c", command],
        capture_output=True,
        text=True,
        timeout=timeout,
        cwd=self.workspace_dir
    )
    return result.stdout + result.stderr
```

### Natural Isolation Properties

- **No persistent shell state** — each command starts fresh (no environment
  variable leakage between actions, no background processes surviving)
- **Working directory scoping** — `cwd` parameter limits default file access
- **Timeout enforcement** — prevents runaway processes
- **No network restrictions** — this is the main weakness

### When This Is Sufficient

For benchmark evaluation (SWE-bench), mini-SWE-agent runs inside a Docker
container provided by the benchmark harness. The Docker container IS the sandbox.
The agent itself does not need sandboxing because the external environment provides
it.

This pattern — "let the deployment environment handle sandboxing" — is common in
simple agents. It works when you control the deployment but is dangerous when
running on a developer's local machine.

## 11. Capy: Full VM Per Task

[Capy](https://github.com/cappyai/capy) takes the most heavyweight approach: every
task gets a complete Ubuntu virtual machine with `sudo` access.

### Design Philosophy

Rather than restricting what the agent can do inside the sandbox, Capy gives the
agent full root access inside a disposable VM. The agent can install packages, modify
system configuration, run services — anything. The VM is destroyed after the task
completes, so nothing persists.

```
  +---------------------------------------------+
  |  Orchestrator                                |
  |  +-------------+  +-------------+            |
  |  |  Task 1 VM  |  |  Task 2 VM  |  ...      |
  |  |  Ubuntu 22  |  |  Ubuntu 22  |            |
  |  |  sudo root  |  |  sudo root  |            |
  |  |  +-------+  |  |  +-------+  |            |
  |  |  | Agent |  |  |  | Agent |  |            |
  |  |  +-------+  |  |  +-------+  |            |
  |  +-------------+  +-------------+            |
  +---------------------------------------------+
```

### Trade-offs

**Advantages:**
- **Maximum freedom** — agent can do anything a developer can
- **Complete isolation** — VMs share nothing (no kernel, no filesystem)
- **Reproducible** — fresh VM means deterministic starting state
- **No sandboxing complexity** — no seccomp rules to maintain

**Disadvantages:**
- **Startup time** — VMs take 10-60 seconds to boot
- **Resource cost** — each VM needs dedicated RAM, CPU, disk
- **Network control** — internet access restricted to package managers only
- **Not practical for interactive use** — too slow for edit-test loops

### When VMs Make Sense

Full VMs are appropriate when:
- Tasks require system-level changes (installing system packages, configuring services)
- Security requirements are extremely high (financial, medical, government)
- Tasks are long-running and the startup cost is amortized
- Multi-user environments where cross-task isolation is critical

## 13. Security vs Usability Trade-off

### The Spectrum

Every sandboxing decision exists on a spectrum between security and usability.
More isolation means more safety but also more friction — slower startups, harder
debugging, incompatible tools, and frustrated developers.

```
  Maximum Security                              Maximum Usability
  <---------------------------------------------------------------->

  Air-gapped VM    Cloud sandbox    Docker    Namespace    No sandbox
  (Capy)           (E2B/Daytona)    (OpenHands) (Codex)    (mini-SWE)

  * No risk         * Low risk      * Medium    * Good      * Full risk
  * 60s startup     * 150ms startup * 5s start  * 1ms start * 0ms
  * Hard to debug   * API overhead  * Docker    * Linux     * Any env
  * Full sudo       * Template mgmt   required    only      * Easy debug
```

### How Agents Find the Right Balance

**Codex's insight:** Layer cheap mechanisms (Landlock + seccomp cost nearly nothing)
and use expensive mechanisms (namespaces) only when needed. The overhead is dominated
by LLM API latency anyway, so sub-second sandbox setup is effectively free.

**OpenHands' insight:** Developers already have Docker installed. Use what is available
rather than requiring exotic kernel features. The startup cost is acceptable for
task-level granularity (one container per conversation, not per command).

**E2B/Daytona's insight:** Push the sandbox to the cloud. The agent framework does not
need to know about Linux security primitives. An API call creates an isolated
environment, and the complexity is someone else's problem.

**Gemini CLI's insight:** Not all operations are equally dangerous. Sandbox the
risky ones (shell commands, code execution), skip sandboxing for safe ones (file reads,
searches). This minimizes overhead while maintaining security where it matters.

### The Trust Hierarchy

Agents implement a hierarchy of trust based on the source of the action:

```
  Most trusted                              Least trusted
  <------------------------------------------------------------>

  User-written     Agent-planned    LLM-generated    External
  config/rules     file edits       shell commands    content
                                                      (prompt
                                                       injection)
```

Sandboxing intensity should scale inversely with trust. User configuration runs
unsandboxed. LLM-generated shell commands run in full sandbox. This adaptive
approach maximizes both security and usability.

### The Future

The trend is toward **transparent sandboxing** — mechanisms so lightweight that
they are always on, with no user-visible overhead. Landlock and seccomp are
approaching this ideal on Linux. As these primitives mature, the trade-off between
security and usability will continue to narrow, eventually making "no sandbox"
an indefensible choice for any agent that executes code.
