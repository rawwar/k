# Test-Driven Development

> How coding agents adopt, adapt, and subvert the Red-Green-Refactor cycle — from
> test-first verification to test-after feedback loops, and why the relationship
> between generated tests and generated code is fundamentally different from human TDD.

---

## TDD in the Agent Context

Traditional test-driven development follows a disciplined cycle: **write a failing test**, **write the minimal code to make it pass**, **refactor**. The developer's intent flows through the test first, creating a specification before an implementation. This works because the test author and the code author share an understanding of correctness but bring different perspectives — the test encodes *what* should happen; the code encodes *how*.

When an AI coding agent enters this picture, the dynamic changes fundamentally. The agent can generate both tests *and* implementation from the same prompt, the same model weights, the same context window. This creates what we call the **dual-authorship problem**: if the same system writes the test and the code, the test may simply validate the agent's misunderstanding rather than catch it.

```
┌─────────────────────────────────────────────────────────────────────┐
│              THE DUAL-AUTHORSHIP PROBLEM                            │
│                                                                     │
│   Human TDD                        Agent TDD                       │
│   ─────────                        ─────────                       │
│   Human writes test ──┐            Agent writes test ──┐           │
│                       │ Different                      │ Same      │
│                       │ perspectives                   │ model     │
│   Human writes code ──┘            Agent writes code ──┘           │
│                                                                     │
│   Test catches bugs because        Test may validate the bug        │
│   test and code encode             because both encode the          │
│   different mental models          same statistical prediction      │
│                                                                     │
│   ✅ Strong verification           ⚠️  Weak verification           │
└─────────────────────────────────────────────────────────────────────┘
```

This doesn't make agent TDD useless — far from it. But it means the value of testing in the agent context comes from different sources than in human TDD:

1. **Existing tests are gold** — Tests written by humans (before the agent runs) provide genuinely independent verification. This is why SWE-bench's methodology works: the gold tests were written by human maintainers, not generated by the agent.

2. **Runtime verification catches what prediction misses** — Even when an agent writes both test and code, *running* the test catches syntax errors, import failures, type mismatches, and runtime exceptions that the model's prediction would gloss over.

3. **Test frameworks enforce structure** — A test that must conform to pytest/jest/go-test conventions imposes constraints the model can't easily hallucinate around.

4. **Regression detection requires no authorship independence** — Running *existing* tests after a code change doesn't require the agent to have written those tests. The value is pure: did the change break something?

## Test-First Approaches

### External Gold Tests (SWE-bench Methodology)

The purest form of test-first in the agent world comes from **benchmark evaluation**. SWE-bench provides each task with a gold-standard test that was written by human maintainers to verify the original fix. The agent never sees the test source — it only receives the issue description and must produce a patch that, when applied, makes the hidden test pass.

This methodology is significant because it completely eliminates the dual-authorship problem. The test encodes human intent; the agent encodes machine understanding. When they agree, there is genuine evidence of correctness.

As documented in [OpenHands](../../agents/openhands/) benchmarks:
> "Each instance is a real GitHub issue + the gold-standard patch that resolved it. The agent receives the issue text and repository state, then must produce a diff."

**Deterministic scoring** follows: patches are applied and the project's existing test suite determines pass/fail — no LLM-based judging. This is TDD in its most rigorous form, even though the agent never explicitly performs a Red-Green-Refactor cycle.

### Agent-Initiated Test-First Patterns

Some agents can be prompted or configured to write tests *before* implementation:

**[Claude Code](../../agents/claude-code/)** follows a three-phase loop — gather context, take action, verify results. For bug fixes, the "gather" phase often includes running existing tests to identify failures. The pattern emerges:

1. `Bash("npm test -- --grep 'session'")` → sees the failing test
2. `Read("src/auth/session.ts")` → understands the code
3. `Edit("src/auth/session.ts", ...)` → fixes the issue
4. `Bash("npm test -- --grep 'session'")` → verifies the fix passes

This is implicitly test-first: the agent identifies the Red state before attempting Green. Anthropic's documentation emphasizes that Claude performs "dramatically better" when it can verify its own work — tests, linters, screenshots. Without verification criteria, the user becomes the only feedback loop.

**[Junie CLI](../../agents/junie-cli/)** goes further, treating test execution as a **first-class operation**, not an afterthought. Its workflow explicitly establishes a baseline before making changes:

```
Junie's Test-First Workflow:
  1. Understand task
  2. Identify relevant tests
  3. Run tests (establish baseline)
     Record: 47 passed, 0 failed, 2 skipped
  4. Modify code
  5. Run tests (verification)
  6. If failures → diagnose → fix → re-run
  7. Return result with test evidence
```

The baseline step is critical — it enables **regression detection** later. If a test that passed in step 3 fails in step 5, Junie knows it introduced a regression and may revert its approach entirely.

### [Codex CLI](../../agents/codex/) — Sandboxed Test-First

Codex executes all commands inside an OS-native sandbox (Seatbelt on macOS, bubblewrap+seccomp on Linux). Test execution follows the same sandboxed path as every other shell command — the filesystem is read-only by default, with only the working directory writable. This means tests run in a controlled environment where the agent cannot accidentally corrupt state.

The non-interactive `codex exec` mode wraps the same core loop for CI contexts, auto-approving tool calls based on policy. This enables test-first workflows in automated pipelines where no human is present to approve each test run.

## Test-After Approaches

### The Default Mode: Implement, Then Verify

Most agents follow a test-after pattern by default. The agent generates an implementation, then runs existing tests to verify correctness. This is not TDD in the traditional sense, but it *is* the pragmatic reality for most agent workflows.

**[Aider](../../agents/aider/)** is the clearest example with its `--auto-test` flag. When enabled, Aider runs the user-specified test suite after every edit:

```
┌─────────────────────────────────────────────────────────────────────┐
│              AIDER'S TEST-AFTER LOOP                                │
│                                                                     │
│  1. User provides task                                              │
│  2. LLM generates code edits                                       │
│  3. Apply edits to files                                            │
│  4. Git commit                                                      │
│  5. Run linter (if --auto-lint)                                     │
│  6. Run test suite (if --auto-test via --test-cmd)                  │
│  7. If tests PASS → done                                            │
│  8. If tests FAIL:                                                  │
│     a. Capture first 50 lines of error output                       │
│     b. Send to LLM: "The tests are correct. Fix the code."         │
│     c. LLM generates fix edits                                      │
│     d. Apply edits → commit → re-test                               │
│     e. Limited retry attempts to prevent infinite loops              │
│                                                                     │
│  This is the same approach used in benchmarks:                      │
│  the LLM gets two attempts (initial + one fix based on test output) │
└─────────────────────────────────────────────────────────────────────┘
```

Configuration is explicit — the user must provide `--test-cmd`:

```bash
# Aider test-after configuration
aider --auto-test --test-cmd "pytest tests/ -x"

# Combined with auto-lint for full verification
aider --auto-test --test-cmd "pytest tests/ -x" \
      --auto-lint --lint-cmd "ruff check ."
```

The key insight from Aider's approach: **the agent is told the tests are correct**. This framing prevents the model from "fixing" the tests instead of the code — a subtle but critical prompt engineering decision.

### [OpenHands](../../agents/openhands/) — Docker-Isolated Test-After

OpenHands runs a full **FastAPI server inside a Docker container** — the `ActionExecutionServer`. Test execution goes through the same sandboxed channel as every other command:

```
┌──────────────────┐          HTTP/REST          ┌─────────────────────┐
│   Host Machine   │  ◄──────────────────────►   │   Docker Container  │
│  DockerRuntime   │     POST /execute_action     │  ActionExecution    │
│  (client)        │     GET  /alive              │  Server (FastAPI)   │
└──────────────────┘                               └─────────────────────┘
```

The agent issues a `CmdRunAction` (e.g., `pytest`), the sandbox executes it, and a `CmdOutputObservation` with stdout/stderr comes back. Full environment fidelity means the agent can run tests, install dependencies, and inspect runtime behavior — all isolated from the host.

### [Gemini CLI](../../agents/gemini-cli/) — Named Test-Driven Pattern

Gemini CLI's tool system documentation explicitly names a "test-driven pattern" in its tool chain examples:

```
read_file(test) → replace(implementation) → run_shell_command("npm test")
```

Its system prompt instructs: "Always run tests after modifications." Auto-approval can be configured for test commands:

```json
{
  "autoApprove": [
    { "tool": "run_shell_command", "pattern": "npm test*" },
    { "tool": "write_file", "pathPattern": "src/**/*.test.ts" }
  ]
}
```

### When Test-After Is Acceptable

The "write code then add tests" pattern is an anti-pattern in human TDD — but for agents, it is often the *only* viable pattern. Several conditions make test-after acceptable:

1. **Existing test suites already cover the code path** — The agent modifies code that is already well-tested. Running existing tests provides genuine verification without needing new tests.

2. **The task is a bug fix with a known reproduction** — The failing behavior is already observable; the agent just needs to make it stop failing.

3. **Framework constraints enforce correctness** — Type-checked languages (Rust, Go, TypeScript with strict mode) catch entire classes of errors at build time, reducing the need for agent-generated tests.

## Red-Green-Refactor with Agents

### The Classical Cycle, Adapted

```
┌─────────────────────────────────────────────────────────────────────┐
│              RED-GREEN-REFACTOR IN AGENT CONTEXT                    │
│                                                                     │
│   ┌───────────┐      ┌───────────┐      ┌───────────┐             │
│   │           │      │           │      │           │             │
│   │    RED    │─────►│   GREEN   │─────►│ REFACTOR  │             │
│   │           │      │           │      │           │             │
│   └───────────┘      └───────────┘      └───────────┘             │
│                                                                     │
│   Agent identifies    Agent writes       Agent cleans up            │
│   or writes a         minimal code       the implementation         │
│   failing test        to pass it                                    │
│                                                                     │
│   ✅ Strong           ✅ Strong          ⚠️  Weak                  │
│   (especially with    (the core          (agents tend to            │
│    existing tests)     strength)          skip this step)           │
└─────────────────────────────────────────────────────────────────────┘
```

### Red: Identifying or Creating the Failing State

The Red phase maps well to agent workflows:

- **Bug fix tasks**: The issue itself describes the Red state. The agent can reproduce it by running existing tests or creating a minimal reproduction script.
- **Feature tasks**: The agent can write tests that exercise the new feature — these fail because the feature doesn't exist yet.
- **SWE-bench**: The gold test is the Red state. The agent never sees it, but the benchmark infrastructure runs it after the patch.

### Green: The Agent's Core Strength

Making a failing test pass is the most natural task for a coding agent. The feedback is binary (pass/fail), the goal is concrete (make this specific test pass), and the iteration cycle is tight (edit → run → check). This is where agents excel.

### Refactor: The Missing Step

In practice, agents almost universally skip the Refactor phase. Several factors explain this:

1. **No explicit trigger** — The Red→Green transition has a clear signal (test passes). The Green→Refactor transition has no equivalent signal — "is this code clean enough?" is subjective.

2. **Token economics** — Refactoring costs tokens without changing test outcomes. In benchmark settings, there is zero incentive to refactor.

3. **Risk of regression** — Refactoring working code risks breaking it. Agents are conservative — once tests pass, they stop.

4. **Missing quality models** — Agents lack the aesthetic sense that drives human developers to refactor. Code that works *is* done, from the model's perspective.

[ForgeCode](../../agents/forgecode/) partially addresses this with its **enforced verification skill** — the runtime blocks task completion unless a verification pass has been performed. The verification asks: "What evidence proves this objective is actually complete?" This is closer to review than refactoring, but it does force the agent to reconsider its work before declaring victory.

### A Complete Red-Green Cycle in Practice

```python
# Agent-driven Red-Green cycle for a bug fix
# Red: the existing test fails
# ──────────────────────────────
# tests/test_user.py
def test_user_email_validation():
    """Regression test: emails with '+' alias should be valid."""
    user = User(email="dev+test@example.com")
    assert user.is_valid()  # Currently fails — bug in regex

# Green: agent generates the minimal fix
# ──────────────────────────────
# src/user.py (before)
EMAIL_REGEX = r'^[a-zA-Z0-9_.]+@[a-zA-Z0-9-]+\.[a-zA-Z]{2,}$'

# src/user.py (after — agent adds '+' to character class)
EMAIL_REGEX = r'^[a-zA-Z0-9_.+]+@[a-zA-Z0-9-]+\.[a-zA-Z]{2,}$'

# Verify: agent runs the test
# $ pytest tests/test_user.py::test_user_email_validation -v
# PASSED ✓

# Refactor: (skipped — agent considers the task complete)
```

## Test Generation Patterns

### Unit Test Generation Across Frameworks

Agents infer the appropriate test framework from project configuration files. The detection chain typically follows:

| Signal | Framework | Language |
|--------|-----------|----------|
| `pytest` in `pyproject.toml` or `requirements.txt` | pytest | Python |
| `unittest` imports in existing tests | unittest | Python |
| `jest` in `package.json` devDependencies | Jest | JavaScript/TypeScript |
| `vitest` in `package.json` or `vite.config.*` | Vitest | JavaScript/TypeScript |
| `go test` convention (`*_test.go` files) | go test | Go |
| `#[test]` attributes in existing code | cargo test | Rust |
| `junit` in `pom.xml` or `build.gradle` | JUnit | Java |

### Framework-Specific Examples

**pytest** — The most common framework in Python agent workflows:

```python
# Agent-generated pytest test with fixtures and parametrize
import pytest
from src.parser import parse_config

@pytest.fixture
def sample_config(tmp_path):
    config_file = tmp_path / "config.yaml"
    config_file.write_text("database:\n  host: localhost\n  port: 5432\n")
    return config_file

@pytest.mark.parametrize("key,expected", [
    ("database.host", "localhost"),
    ("database.port", 5432),
    ("database.missing", None),
])
def test_parse_config_nested_keys(sample_config, key, expected):
    config = parse_config(sample_config)
    assert config.get(key) == expected

def test_parse_config_invalid_yaml(tmp_path):
    bad_file = tmp_path / "bad.yaml"
    bad_file.write_text("invalid: yaml: content: [")
    with pytest.raises(ValueError, match="Invalid YAML"):
        parse_config(bad_file)
```

**Jest** — The dominant framework in JavaScript/TypeScript agent work:

```typescript
// Agent-generated Jest test with mocks and async handling
import { fetchUserProfile } from '../src/api/users';
import { httpClient } from '../src/http';

jest.mock('../src/http');
const mockGet = httpClient.get as jest.MockedFunction<typeof httpClient.get>;

describe('fetchUserProfile', () => {
  afterEach(() => jest.resetAllMocks());

  it('returns parsed user profile for valid ID', async () => {
    mockGet.mockResolvedValueOnce({
      data: { id: 42, name: 'Ada Lovelace', email: 'ada@example.com' },
    });

    const profile = await fetchUserProfile(42);

    expect(mockGet).toHaveBeenCalledWith('/api/users/42');
    expect(profile).toEqual({
      id: 42,
      name: 'Ada Lovelace',
      email: 'ada@example.com',
    });
  });

  it('throws NotFoundError for non-existent user', async () => {
    mockGet.mockRejectedValueOnce({ status: 404 });

    await expect(fetchUserProfile(999)).rejects.toThrow('User not found');
  });
});
```

### [Junie CLI](../../agents/junie-cli/) — Structured TestResult Objects

Junie CLI stands apart by providing **structured test results** rather than raw console output. In IDE mode, the test runner returns rich objects:

```kotlin
// IDE provides structured test results
data class TestResult(
    val name: String,
    val status: TestStatus,    // PASSED, FAILED, ERROR, SKIPPED
    val duration: Duration,
    val failureMessage: String?,
    val stackTrace: List<StackFrame>?,
    val sourceLocation: PsiElement?,  // Direct link to source
    val diffOutput: String?           // Expected vs actual
)
```

In CLI mode, Junie parses raw output into an equivalent structure:

```
Parsed structure:
  TestResult {
    test_name: "test_create_user"
    file: "tests/test_user.py"
    line: 42
    status: FAILED
    error_type: "AssertionError"
    message: "assert 400 == 201"
    expected: 201
    actual: 400
  }
```

This structured representation gives the LLM far more actionable information than a wall of console text. The agent knows exactly which test failed, where in the source it lives, and what the expected-vs-actual difference was.

### [ForgeCode](../../agents/forgecode/) — Dynamic Skill Loading for Tests

ForgeCode's architecture routes tasks through a **skill engine**. When the task profile indicates test writing, the testing skill is loaded dynamically:

> "A test-writing task loads the testing skill. A debugging task loads the debugging skill. A refactoring task loads the refactoring skill. This keeps the system prompt lean and relevant. Skills that aren't needed don't consume context."

This is context management at the instruction level — the system prompt is tailored to the task type, so a test-generation task gets specialized instructions about assertion patterns, edge cases, and framework conventions without competing for token budget with irrelevant debugging or refactoring instructions.

## Framework-Specific Integration

### Detection: How Agents Know Which Framework to Use

Agents determine the test framework through a priority chain of signals:

1. **Existing test files** — If `tests/test_*.py` files exist with `import pytest`, use pytest. If `*.test.ts` files exist with `describe/it`, use Jest or Vitest.
2. **Configuration files** — `pyproject.toml` (`[tool.pytest]`), `package.json` (`"jest"` or `"vitest"` in scripts/devDeps), `Cargo.toml`, `pom.xml`.
3. **Lock files and dependency manifests** — `requirements.txt`, `package-lock.json`, `go.sum`.
4. **Language conventions** — Go files ending in `_test.go`, Rust `#[cfg(test)]` modules.

[Gemini CLI](../../agents/gemini-cli/) auto-approves test commands based on pattern matching — `"npm test*"` matches any npm test invocation, reducing friction for test-after workflows.

[Claude Code](../../agents/claude-code/) supports test command allowlisting in its permission configuration:

```json
{
  "allow": [
    "Bash(npm run lint)",
    "Bash(npm run test *)",
    "Bash(pytest *)"
  ]
}
```

This pre-approves test execution, removing the human-in-the-loop bottleneck for verification steps.

### Language-Specific Patterns

**Go — Table-Driven Tests**: Agents generating Go tests naturally produce table-driven patterns because Go's testing convention strongly favors them and training data reinforces the pattern:

```go
func TestParseConfig(t *testing.T) {
    tests := []struct {
        name    string
        input   string
        want    Config
        wantErr bool
    }{
        {
            name:  "valid minimal config",
            input: `{"host": "localhost", "port": 5432}`,
            want:  Config{Host: "localhost", Port: 5432},
        },
        {
            name:    "invalid JSON",
            input:   `{invalid}`,
            wantErr: true,
        },
        {
            name:    "missing required field",
            input:   `{"host": "localhost"}`,
            wantErr: true,
        },
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := ParseConfig([]byte(tt.input))
            if (err != nil) != tt.wantErr {
                t.Errorf("ParseConfig() error = %v, wantErr %v", err, tt.wantErr)
                return
            }
            if !tt.wantErr && got != tt.want {
                t.Errorf("ParseConfig() = %v, want %v", got, tt.want)
            }
        })
    }
}
```

**Rust — `#[test]` and `#[should_panic]`**: Rust's built-in test attributes produce compact, zero-dependency tests that agents can generate reliably.

**Java — JUnit 5 with Maven Surefire**: The Maven ecosystem's test discovery and Surefire plugin reporting give agents structured XML output to parse.

## Test Output Parsing

### The Truncation Insight

One of the most counterintuitive findings in agent testing comes from [Aider's](../../agents/aider/) benchmark work: **50 lines of test output is optimal**. More output *hurts* performance.

From Aider's benchmark methodology:
> "If tests fail, send first 50 lines of error output + 'fix it' message."

This truncation serves several purposes:

1. **Focuses attention** — The first failure is usually the most informative. Cascading failures from a single root cause generate noise.
2. **Preserves context budget** — Sending 500 lines of test output into the context window displaces useful context (source code, instructions, history).
3. **Reduces confusion** — Models can get "lost" in long error traces, attempting to fix secondary issues while missing the root cause.

### Structured vs Raw Output

The spectrum of test output handling across agents reveals a maturity gradient:

```
┌─────────────────────────────────────────────────────────────────────┐
│              TEST OUTPUT PARSING SPECTRUM                            │
│                                                                     │
│   Raw stdout/stderr        Truncated raw         Structured parse   │
│   ◄────────────────────────────────────────────────────────────────► │
│                                                                     │
│   mini-SWE-agent           Aider (50 lines)      Junie CLI          │
│   OpenCode                 Claude Code            ForgeCode          │
│   Goose                    Codex CLI              Droid              │
│   Sage Agent               Gemini CLI                                │
│                            OpenHands                                 │
│                                                                     │
│   Low signal-to-noise      Balanced               High signal        │
│   High token cost          Practical              Low token cost     │
└─────────────────────────────────────────────────────────────────────┘
```

### Extracting Actionable Information

Effective test output parsing extracts four key fields:

| Field | Example | Why It Matters |
|-------|---------|----------------|
| **File path** | `tests/test_user.py` | Agent knows which test file to examine |
| **Line number** | `:42` | Agent can jump to the exact assertion |
| **Assertion message** | `assert 400 == 201` | Reveals expected vs actual |
| **Stack trace** | `src/api.py:87 → src/validator.py:23` | Shows the call chain to the failure |

[Junie CLI](../../agents/junie-cli/) extracts all four in structured form. [Aider](../../agents/aider/) relies on the model to parse them from truncated raw output. The tradeoff is implementation complexity vs. robustness — structured parsing requires framework-specific parsers but produces more reliable agent behavior.

## The Test Feedback Loop

### The Core Iteration Pattern

Every agent that uses tests follows a variation of the same feedback loop:

```python
# Pseudocode: Agent test-run-fix loop
def agent_test_feedback_loop(task, max_retries=3):
    # Phase 1: Understand and implement
    context = gather_context(task)
    implementation = llm_generate_code(context)
    apply_edits(implementation)

    # Phase 2: Test and iterate
    for attempt in range(max_retries):
        test_result = run_tests()

        if test_result.all_passed:
            commit_changes()
            return Success(test_result)

        # Parse failures and feed back to LLM
        error_summary = truncate(test_result.stderr, max_lines=50)
        fix_prompt = f"""
        The tests are correct. Fix the code to resolve these errors:
        {error_summary}
        """
        fix = llm_generate_fix(context + fix_prompt)
        apply_edits(fix)

    return Failure("Max retries exceeded", test_result)
```

### Bounded Retries: Finding the Sweet Spot

Different agents use different retry limits, and the numbers reveal sharp diminishing returns:

| Agent | Max Retries | Rationale |
|-------|-------------|-----------|
| [Aider](../../agents/aider/) | 2 (initial + 1 fix) | Benchmark standard; mirrors real usage |
| [Junie CLI](../../agents/junie-cli/) | 3–5 | Configurable; escalates to user after limit |
| [Claude Code](../../agents/claude-code/) | Unbounded (user-directed) | Model decides when to stop; user can interrupt |
| [ForgeCode](../../agents/forgecode/) | Runtime-enforced verification | Must call verification skill before completion |
| [mini-SWE-agent](../../agents/mini-swe-agent/) | Fixed iteration limit | System prompt guides workflow |

Aider's 2-attempt structure deserves emphasis. From their benchmark documentation:
> "Two attempts: Models get one try, plus one fix attempt with test output. This mirrors real usage (write code, debug based on errors)."

This is a **fair approximation** of interactive use. In practice, a human developer would typically give an agent one shot, review the failure, and either fix it themselves or give the agent one more chance with the error context. Going beyond two attempts often indicates the model doesn't understand the problem rather than needing more iterations.

### Regression Detection

[Junie CLI](../../agents/junie-cli/) implements the most sophisticated regression detection among the agents studied. Its verification phase distinguishes four scenarios:

```
Before changes:
  Run existing tests → Establish baseline
  Record: 47 passed, 0 failed, 2 skipped

After changes:
  Run tests again → Check for regression

  Scenario A: All pass (47 + 3 new) → ✅ Success! Present results
  Scenario B: New tests fail         → Fix implementation, re-run
  Scenario C: Existing tests fail    → ⚠️ Regression! Analyze and fix
               → May need to revert approach
  Scenario D: Build fails            → Fix syntax/type issues, re-run
```

Scenario C is the critical one: "If previously passing tests start failing, the agent may roll back changes and reassess." This is qualitatively different from simply re-running tests — it implies the agent's approach was wrong, not just its implementation.

### [Ante](../../agents/ante/) — Role-Restricted Test Execution

[Ante](../../agents/ante/) takes an architectural approach to test feedback: sub-agents have **role-restricted tool access**. A sub-agent writing code gets file system and search tools but not shell execution. A sub-agent running tests gets shell execution but not file writes. This separation enforces a clean boundary between code generation and verification.

## Benchmark Evaluation and TDD

### SWE-bench: The Gold Standard

SWE-bench's evaluation methodology is fundamentally test-driven:

1. Agent receives issue description + repository state
2. Agent produces a patch (diff)
3. Patch is applied to the repository
4. **Gold tests are run** — tests written by human maintainers
5. Binary result: all tests pass or fail

The agent never sees the test source code, though the tests were likely in training data. Temperature is set to 0 for reproducibility. This creates a clean separation: the agent's job is to understand the *intent* of the issue and produce code that satisfies it, not to satisfy a test it can read and reverse-engineer.

### The Verification–Performance Correlation

Evidence from multiple agents converges on a stark finding: **verification loops are the single largest lever for benchmark performance**. Agents without test feedback score 30–40% on SWE-bench. The same agents with Edit-Apply-Verify cycles reach 60–80%+.

The relationship is not merely correlational. [ForgeCode's](../../agents/forgecode/) enforced verification skill was described as "the biggest single improvement" — GPT-based models would implement solutions, sound confident, and stop before the task was actually complete. Enforcement caught these premature completions.

### Terminal-Bench and Task-Oriented Verification

Beyond SWE-bench, newer benchmarks like Terminal-Bench 2.0 use task-oriented verification: the success criterion is whether the agent accomplished a terminal-based task, verified by checking system state (file contents, process status, network configuration) rather than test suites. This expands the notion of "testing" beyond code correctness into operational verification.

## Design Recommendations

1. **Prefer existing tests over generated tests for verification.** Human-written tests encode intent that is independent of the agent's model. When existing tests are available, they provide stronger verification than any test the agent could generate.

2. **Truncate test output aggressively.** Aider's 50-line rule is counterintuitive but well-supported — more error output hurts more than it helps. The first failure contains the root cause; cascading failures are noise.

3. **Bound your retry loop at 2–3 attempts.** Diminishing returns are sharp. If the agent hasn't fixed it in 3 attempts, it needs a different approach (or human intervention), not another retry.

4. **Establish baselines before making changes.** Junie CLI's pattern of running tests *before* editing code enables regression detection that simple test-after approaches miss entirely.

5. **Structure test output when possible.** Parsing test results into file/line/assertion/stack-trace fields gives the LLM dramatically more actionable information than raw console output.

6. **Enforce verification programmatically.** ForgeCode's insight — that prompting "please verify" doesn't work but runtime enforcement does — applies broadly. If verification matters, make it mandatory, not optional.

7. **Pre-approve test commands.** Claude Code's permission allowlisting and Gemini CLI's auto-approve patterns remove friction from the feedback loop without compromising security, since test commands are read-only by nature.

8. **Separate code generation from test execution roles.** Ante's pattern of role-restricted sub-agents prevents the dual-authorship problem at the architectural level.
