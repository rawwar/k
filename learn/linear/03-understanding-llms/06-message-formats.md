# Message Formats

> **What you'll learn:**
> - How messages are structured with roles (system, user, assistant) and content blocks
> - The differences between Anthropic and OpenAI message format conventions
> - How tool use messages, tool results, and multi-turn conversations are represented in the message array

Every API call to an LLM includes a messages array -- the conversation history that the model uses to generate its response. Understanding the exact structure of this array is essential because your agent constructs it on every iteration of the agentic loop. Get the format wrong and the API rejects your request. Get the structure suboptimal and the model makes poor decisions because it misunderstands the conversation state.

## The Messages Array

Both Anthropic and OpenAI structure conversations as an ordered array of message objects. Each message has a **role** and **content**. The messages must alternate between `user` and `assistant` roles (with some exceptions for tool results).

Here is the simplest possible conversation:

```json
{
  "messages": [
    {"role": "user", "content": "What is the Rust equivalent of Python's dict?"}
  ]
}
```

And a multi-turn conversation:

```json
{
  "messages": [
    {"role": "user", "content": "What is the Rust equivalent of Python's dict?"},
    {"role": "assistant", "content": "The Rust equivalent is `HashMap` from the `std::collections` module."},
    {"role": "user", "content": "How do I iterate over it?"}
  ]
}
```

The model receives the entire array and generates the next assistant response. It does not have memory between API calls -- your agent is responsible for accumulating messages and sending the full history every time.

## Roles

### User

The `user` role represents messages from the human (or, in an agent context, messages composed by the agent on behalf of the user). User messages are the primary input that drives the model's response.

### Assistant

The `assistant` role represents messages previously generated by the model. By including assistant messages in the history, you give the model context about what it has already said and done. This is critical for multi-turn agents -- without the assistant's previous responses, the model has no idea what tools it already called or what code it already wrote.

### System (OpenAI) / System Field (Anthropic)

The system role works differently between providers:

**Anthropic** places the system prompt in a top-level `system` field, separate from the messages array:

```json
{
  "model": "claude-sonnet-4-20250514",
  "system": "You are an expert Rust coding assistant.",
  "messages": [
    {"role": "user", "content": "Help me fix this error"}
  ]
}
```

**OpenAI** places the system prompt as the first message with `role: "system"`:

```json
{
  "model": "gpt-4o",
  "messages": [
    {"role": "system", "content": "You are an expert Rust coding assistant."},
    {"role": "user", "content": "Help me fix this error"}
  ]
}
```

When building a provider-agnostic agent, you will store the system prompt separately in your internal data model and inject it in the correct location based on which provider you are calling.

## Content Blocks (Anthropic)

Anthropic's API supports a richer content format where the `content` field can be either a plain string or an array of content blocks. This is how Anthropic represents multi-part messages, including tool use:

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "I'll read the file first to understand the current implementation."
    },
    {
      "type": "tool_use",
      "id": "toolu_01A09q90qw90lq917835lq9",
      "name": "read_file",
      "input": {"path": "src/main.rs"}
    }
  ]
}
```

This is a single assistant message that contains both text and a tool call. The content block format lets the model express "here is my reasoning, and here is the action I am taking" in one message.

The content block types you will encounter in Anthropic's API:

| Type | Used In | Purpose |
|---|---|---|
| `text` | user, assistant | Plain text content |
| `tool_use` | assistant | Model requests a tool call |
| `tool_result` | user | Agent returns tool execution result |
| `image` | user | Image input (base64 or URL) |

## Tool Use Message Flow

The most important message pattern for agents is the tool use cycle. Here is the complete flow using Anthropic's format:

**Turn 1: User sends a request**
```json
{"role": "user", "content": "What files are in the src directory?"}
```

**Turn 2: Assistant responds with a tool call**
```json
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "Let me check the contents of the src directory."
    },
    {
      "type": "tool_use",
      "id": "toolu_01XFDUDYJgAACzvnptvVer6C",
      "name": "shell",
      "input": {"command": "ls src/"}
    }
  ]
}
```

**Turn 3: User role message containing the tool result**
```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01XFDUDYJgAACzvnptvVer6C",
      "content": "main.rs\nlib.rs\ntools/\nconfig.rs"
    }
  ]
}
```

**Turn 4: Assistant processes the result and responds**
```json
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "The src directory contains:\n- `main.rs` - the entry point\n- `lib.rs` - the library root\n- `tools/` - a directory for tool implementations\n- `config.rs` - configuration handling"
    }
  ]
}
```

Notice that the tool result is sent as a `user` role message. This maintains the alternating user/assistant pattern. The `tool_use_id` links the result back to the specific tool call, which is important when the model makes multiple tool calls in one turn.

::: python Coming from Python
In Python, you might represent conversation history as a list of dictionaries: `messages = [{"role": "user", "content": "..."}]`. In Rust, you will define an enum for roles and a struct for messages. The Anthropic content block format maps well to Rust enums -- a `ContentBlock` enum with variants for `Text`, `ToolUse`, and `ToolResult` is a natural representation.
:::

## OpenAI's Tool Message Format

OpenAI handles tool use messages differently. Instead of content blocks, tool calls appear as a separate field on the assistant message, and tool results use a dedicated `tool` role:

**Assistant message with tool call:**
```json
{
  "role": "assistant",
  "content": null,
  "tool_calls": [
    {
      "id": "call_abc123",
      "type": "function",
      "function": {
        "name": "shell",
        "arguments": "{\"command\": \"ls src/\"}"
      }
    }
  ]
}
```

**Tool result message:**
```json
{
  "role": "tool",
  "tool_call_id": "call_abc123",
  "content": "main.rs\nlib.rs\ntools/\nconfig.rs"
}
```

Key differences from Anthropic:
- Tool calls are in a `tool_calls` array on the assistant message, not in content blocks
- The `arguments` field is a JSON string, not a parsed object
- Tool results use `role: "tool"` instead of being wrapped in a user message
- The linking field is `tool_call_id` instead of `tool_use_id`

## Message Ordering Rules

Both APIs enforce strict ordering rules:

1. **Messages must alternate between user and assistant** (with tool results being considered part of the user turn in Anthropic, or allowed between assistant and user turns in OpenAI)
2. **The first message must be from the user** (after the system message, if using OpenAI format)
3. **Every tool call must have a corresponding tool result** -- if the assistant requests a tool call, the next message must contain the result for that call
4. **Tool results must reference valid tool call IDs** -- the `tool_use_id` or `tool_call_id` must match an ID from a preceding assistant message

Violating these rules produces API errors that can be confusing if you do not understand the expected structure. Your agent's message management code needs to enforce these invariants.

## Building the Provider Abstraction

Since Anthropic and OpenAI use different message formats, a well-designed agent defines an internal message format and converts to the provider-specific format at the API boundary:

```rust
// Your internal representation
enum Role {
    User,
    Assistant,
    System,
}

enum ContentBlock {
    Text(String),
    ToolUse { id: String, name: String, input: serde_json::Value },
    ToolResult { tool_use_id: String, content: String },
}

struct Message {
    role: Role,
    content: Vec<ContentBlock>,
}
```

Then you write serialization functions that convert `Message` to the Anthropic or OpenAI JSON format. This abstraction is critical for a provider-agnostic agent and is something we will build in detail in later chapters.

::: wild In the Wild
OpenCode, an open-source coding agent, defines a unified message format internally and implements provider-specific serializers for Anthropic and OpenAI. This allows the core agent loop to work identically regardless of which provider is configured. Claude Code, being an Anthropic product, works directly with the Anthropic message format but still maintains clean internal abstractions for message construction.
:::

## Key Takeaways

- The messages array is an ordered sequence of role-tagged messages that your agent constructs and sends on every API call -- the model has no memory between calls
- Anthropic uses content blocks (arrays of typed objects) within messages to represent mixed content like text alongside tool calls, while OpenAI uses separate fields and a dedicated `tool` role
- The tool use message cycle follows a strict pattern: user request, assistant tool call, tool result (linked by ID), assistant follow-up -- violating this pattern produces API errors
- Building a provider-agnostic agent requires an internal message representation that is converted to provider-specific JSON at the API boundary
- Every tool call must have a corresponding tool result with a matching ID, and messages must maintain the alternating role pattern each API expects
