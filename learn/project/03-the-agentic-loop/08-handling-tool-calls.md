# Handling Tool Calls

> **What you'll learn:**
> - How to extract tool_use blocks from an assistant message and parse the tool name and input JSON
> - How to route tool calls to the appropriate handler (placeholder for now, implemented fully in Chapter 4)
> - How to handle the case where the model requests multiple tool calls in a single response

When the model's response has `stop_reason: "tool_use"`, it is asking your agent to do something. The response contains one or more `tool_use` content blocks, each specifying a tool name, a unique ID, and a JSON input. Your job is to extract these requests, execute the corresponding tools, and collect the results. Let's build the machinery for this.

## Anatomy of a Tool Call

A tool-use content block from the API looks like this:

```json
{
    "type": "tool_use",
    "id": "toolu_01ABC123",
    "name": "read_file",
    "input": {
        "path": "src/main.rs"
    }
}
```

Three fields matter:

- **`id`**: A unique identifier generated by the API. You *must* include this same ID in the tool result you send back. This is how the model matches results to requests.
- **`name`**: The tool the model wants to call. This must match a tool registered in your system.
- **`input`**: A JSON object containing the arguments for the tool. The structure depends on the tool's schema.

In your Rust types, this is already represented by the `ContentBlock::ToolUse` variant:

```rust
ContentBlock::ToolUse {
    id: String,       // "toolu_01ABC123"
    name: String,     // "read_file"
    input: Value,     // {"path": "src/main.rs"}
}
```

## Extracting Tool Calls

The first step in handling tool calls is to extract all `ToolUse` blocks from the response. Remember, a single response can contain multiple tool calls -- the model might want to read several files at once, or read a file and run a command in the same turn.

```rust
/// A parsed tool call ready for execution.
#[derive(Debug, Clone)]
pub struct ToolCall {
    /// The unique ID from the API (must be echoed back in the result).
    pub id: String,
    /// The name of the tool to execute.
    pub name: String,
    /// The input arguments as a JSON value.
    pub input: Value,
}

/// Extract all tool calls from a list of content blocks.
pub fn extract_tool_calls(content: &[ContentBlock]) -> Vec<ToolCall> {
    content
        .iter()
        .filter_map(|block| match block {
            ContentBlock::ToolUse { id, name, input } => {
                Some(ToolCall {
                    id: id.clone(),
                    name: name.clone(),
                    input: input.clone(),
                })
            }
            _ => None,
        })
        .collect()
}
```

Using `filter_map` is idiomatic Rust for "filter to a specific variant and extract data." It combines the filtering and mapping into a single pass. The result is a clean `Vec<ToolCall>` with no content block wrapping.

## Processing Multiple Tool Calls

When the model returns multiple tool calls in a single response, you need to execute *all* of them before feeding results back. The API requires that for every `tool_use` block in the assistant's message, there must be a corresponding `tool_result` block in the next user message.

```rust
impl Agent {
    /// Process all tool calls from an assistant response.
    /// Returns a list of tool result content blocks, one per tool call.
    async fn handle_tool_calls(
        &self,
        content: &[ContentBlock],
    ) -> Vec<ContentBlock> {
        let tool_calls = extract_tool_calls(content);
        let mut results = Vec::with_capacity(tool_calls.len());

        for call in &tool_calls {
            let result = self.execute_single_tool(call).await;
            results.push(result);
        }

        results
    }

    /// Execute a single tool call and return the result as a ContentBlock.
    async fn execute_single_tool(
        &self,
        call: &ToolCall,
    ) -> ContentBlock {
        // Dispatch to the appropriate tool handler.
        // For now, this is a stub. Chapter 4 replaces it with
        // a real tool registry.
        let (output, is_error) = match call.name.as_str() {
            // Placeholder implementations for illustration
            "read_file" => {
                let path = call.input["path"]
                    .as_str()
                    .unwrap_or("<missing path>");
                (
                    format!("[Stub] Would read file: {}", path),
                    true,
                )
            }
            "write_file" => {
                let path = call.input["path"]
                    .as_str()
                    .unwrap_or("<missing path>");
                (
                    format!("[Stub] Would write file: {}", path),
                    true,
                )
            }
            unknown => {
                (
                    format!("Unknown tool: '{}'", unknown),
                    true,
                )
            }
        };

        ContentBlock::ToolResult {
            tool_use_id: call.id.clone(),
            content: output,
            is_error: Some(is_error),
        }
    }
}
```

Notice how each tool result includes `tool_use_id: call.id.clone()`. This links the result back to the specific request. If the model made three tool calls with IDs `a`, `b`, `c`, you must return three results with those same IDs in the same order.

::: python Coming from Python
In Python with the Anthropic SDK, you would handle tool calls similarly:
```python
def handle_tool_calls(self, content):
    results = []
    for block in content:
        if block.type == "tool_use":
            output = self.execute_tool(block.name, block.input)
            results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": output,
            })
    return results
```
The pattern is identical. The Rust version uses pattern matching to extract `ToolUse` blocks, while Python checks `block.type`. The Rust version also forces you to handle the `is_error` field explicitly, while in Python you could forget to set it and the API would default to `false`.
:::

## Parallel vs Sequential Execution

When the model requests multiple tool calls, should you execute them in parallel or sequentially? There is a trade-off:

**Sequential** (what we implement here): Simpler, easier to debug. Each tool runs one at a time. If one fails, you still execute the rest (the model expects results for all of them).

**Parallel**: Faster for independent tool calls (reading three different files). But you need to handle the case where tools have side effects that interact (writing to a file, then reading it).

For now, sequential execution is the right choice. Here is how you would add parallel execution later using `tokio::join!` or `futures::future::join_all`:

```rust
use futures::future::join_all;

/// Execute all tool calls in parallel.
/// Use this when tools are independent (e.g., reading different files).
async fn handle_tool_calls_parallel(
    &self,
    content: &[ContentBlock],
) -> Vec<ContentBlock> {
    let tool_calls = extract_tool_calls(content);

    let futures: Vec<_> = tool_calls
        .iter()
        .map(|call| self.execute_single_tool(call))
        .collect();

    join_all(futures).await
}
```

::: wild In the Wild
Claude Code executes tool calls sequentially by default. This is because many tool calls in a coding context have dependencies -- you read a file, modify it, then check if it compiles. Running them in parallel could cause race conditions. OpenCode takes a similar approach: sequential by default with no parallel option. The simplicity and correctness outweigh the performance benefit for most coding tasks.
:::

## Input Validation

Before executing a tool, you should validate that the input matches what the tool expects. The model *usually* sends well-formed JSON that matches the schema you described to it, but "usually" is not "always." Defensive validation prevents crashes:

```rust
/// Validate that a tool call's input contains the required fields.
/// Returns an error message if validation fails, or None if valid.
fn validate_tool_input(
    tool_name: &str,
    input: &Value,
) -> Option<String> {
    match tool_name {
        "read_file" => {
            if !input.get("path").and_then(|v| v.as_str()).is_some() {
                return Some(
                    "read_file requires a 'path' field (string)".to_string()
                );
            }
        }
        "write_file" => {
            if !input.get("path").and_then(|v| v.as_str()).is_some() {
                return Some(
                    "write_file requires a 'path' field (string)".to_string()
                );
            }
            if !input.get("content").and_then(|v| v.as_str()).is_some() {
                return Some(
                    "write_file requires a 'content' field (string)".to_string()
                );
            }
        }
        _ => {
            // Unknown tools are handled by the dispatcher
        }
    }
    None
}
```

Integrate this into the execution path:

```rust
async fn execute_single_tool(
    &self,
    call: &ToolCall,
) -> ContentBlock {
    // Validate input before execution
    if let Some(error_msg) = validate_tool_input(&call.name, &call.input) {
        return ContentBlock::ToolResult {
            tool_use_id: call.id.clone(),
            content: format!("Input validation error: {}", error_msg),
            is_error: Some(true),
        };
    }

    // Proceed with execution...
    // (dispatch code here)
    ContentBlock::ToolResult {
        tool_use_id: call.id.clone(),
        content: "[Stub] Tool execution not yet implemented".to_string(),
        is_error: Some(true),
    }
}
```

Returning a validation error as a `ToolResult` with `is_error: true` is important. The model *needs* to see the error so it can adjust its next call. If you silently ignore bad input, the model has no way to know it did something wrong.

## The Complete Tool Call Flow

Here is the full sequence from response to results, showing how all the pieces fit together:

```rust
// Step 1: The LLM returns a response with tool calls
// response.content = [
//     Text { text: "I'll read those files for you." },
//     ToolUse { id: "toolu_01", name: "read_file", input: {"path": "src/main.rs"} },
//     ToolUse { id: "toolu_02", name: "read_file", input: {"path": "src/lib.rs"} },
// ]

// Step 2: Extract tool calls
let calls = extract_tool_calls(&response.content);
// calls = [
//     ToolCall { id: "toolu_01", name: "read_file", input: {"path": "src/main.rs"} },
//     ToolCall { id: "toolu_02", name: "read_file", input: {"path": "src/lib.rs"} },
// ]

// Step 3: Execute each tool and collect results
let results = agent.handle_tool_calls(&response.content).await;
// results = [
//     ToolResult { tool_use_id: "toolu_01", content: "[file contents]", is_error: None },
//     ToolResult { tool_use_id: "toolu_02", content: "[file contents]", is_error: None },
// ]

// Step 4: Feed results back as a user message
state.add_tool_results(results);
// The next API call will include these results, and the model
// will see the file contents and decide what to do next.
```

The loop does not need to understand what the tools do. It just extracts calls, delegates execution, and feeds results back. This is the abstraction boundary that makes the architecture extensible.

## Key Takeaways

- Tool calls are extracted from `ContentBlock::ToolUse` variants using `filter_map`, producing a clean `Vec<ToolCall>` for processing
- Every tool result must include the matching `tool_use_id` from the original request -- this is how the model correlates results with its requests
- Multiple tool calls in a single response must all be executed: the API expects a result for every tool-use block in the assistant's message
- Input validation before execution prevents crashes and gives the model useful error messages through `is_error: true` results
- Sequential execution is simpler and safer for coding tasks where tool calls may have dependencies; parallel execution can be added later for independent operations
