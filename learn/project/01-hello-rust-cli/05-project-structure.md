# Project Structure

> **What you'll learn:**
> - How to lay out `src/main.rs`, `src/lib.rs`, and supporting modules for a CLI application
> - The role of `Cargo.toml`, `Cargo.lock`, and the `target/` directory in a Rust project
> - How to structure your coding-agent project so it scales as you add features in later chapters

A well-organized project structure saves you hours of confusion later. When your coding agent grows to include an LLM client, tool execution, context management, and a terminal UI, you need to know exactly where each piece lives. Rust has strong conventions for project layout, and Cargo enforces many of them automatically.

Let's look at what you have now and plan where it is going.

## The Default Layout

After running `cargo new kodai`, you have this structure:

```
kodai/
  Cargo.toml          # Project manifest (name, version, dependencies)
  Cargo.lock          # Exact dependency versions (generated by Cargo)
  .gitignore          # Ignores target/ directory
  src/
    main.rs           # Binary entry point
  target/             # Compiled artifacts (generated, never commit)
    debug/
      kodai           # Your compiled binary
```

This is the minimal layout for a binary project. Let's understand each piece.

### Cargo.toml — The Single Source of Truth

`Cargo.toml` is the project manifest. It tells Cargo your project's name, version, Rust edition, and dependencies. As the project grows, you also add build metadata, feature flags, and compiler settings here.

```toml
[package]
name = "kodai"
version = "0.1.0"
edition = "2021"

[dependencies]
```

Every Rust project has exactly one `Cargo.toml` at the root. It is the equivalent of combining Python's `pyproject.toml`, `setup.py`, and `requirements.txt` into a single file.

### Cargo.lock — Reproducible Builds

When Cargo resolves dependencies, it records the exact versions in `Cargo.lock`. This file ensures that everyone who builds your project gets identical dependency versions.

**For binary projects (applications), commit `Cargo.lock` to Git.** This guarantees reproducible builds. For library crates, the convention is to not commit it — but since we are building a CLI application, always commit it.

### src/main.rs — The Binary Entry Point

Cargo looks for `src/main.rs` to build a binary. The `main` function inside it is the program's entry point. You do not configure this anywhere — Cargo uses this convention automatically.

### target/ — Build Artifacts

Everything Cargo compiles goes into `target/`. This directory is in `.gitignore` by default and can grow to hundreds of megabytes. You never need to look inside it except to find your compiled binary at `target/debug/kodai` (or `target/release/kodai` for release builds).

::: python Coming from Python
A Python project might scatter configuration across `pyproject.toml`, `setup.cfg`, `requirements.txt`, `MANIFEST.in`, and `setup.py`. A Rust project puts everything in `Cargo.toml`. The `target/` directory is like `__pycache__`, `.eggs`, and `dist/` combined — all build artifacts in one place that you never commit.
:::

## Adding src/lib.rs — The Library Root

As your project grows, you do not want everything in `main.rs`. Rust projects conventionally split into two parts:

- **`src/main.rs`** — the binary entry point. It handles startup, CLI parsing, and calls into the library.
- **`src/lib.rs`** — the library root. It contains your actual application logic, organized into modules.

Create `src/lib.rs`:

```rust
pub fn greet(name: &str) -> String {
    format!("Hello, {name}! Welcome to Kodai.")
}
```

Now update `src/main.rs` to use it:

```rust
use kodai::greet;

fn main() {
    let message = greet("developer");
    println!("{message}");
}
```

Run `cargo run` and you see:

```
Hello, developer! Welcome to Kodai.
```

The `use kodai::greet` line imports the `greet` function from your library crate. The library's crate name matches the `name` field in `Cargo.toml` (which is `kodai`). The `pub` keyword on `greet` makes it public — visible outside the module. Without `pub`, the function would be private and `main.rs` could not access it.

This split has a practical benefit: code in `src/lib.rs` can be tested independently, imported by integration tests, and eventually reused if you want to expose your agent as a library.

## Planning for Growth

By the end of this book, your coding agent has many components. Here is the structure you are working toward:

```
kodai/
  Cargo.toml
  Cargo.lock
  src/
    main.rs              # Entry point: CLI parsing, startup
    lib.rs               # Library root: re-exports modules
    repl.rs              # The REPL loop
    agent/
      mod.rs             # Agent module root
      loop.rs            # The agentic loop
      context.rs         # Context/message management
    llm/
      mod.rs             # LLM client module root
      client.rs          # HTTP client for API calls
      streaming.rs       # Server-sent events parsing
    tools/
      mod.rs             # Tool registry and dispatch
      shell.rs           # Shell command execution
      file_read.rs       # File reading tool
      file_write.rs      # File writing tool
  tests/
    integration_test.rs  # Integration tests
```

You do not need to create all of these now. The point is to see where the project is headed. Each directory under `src/` is a *module* — Rust's way of organizing code into namespaces. You learn how to create modules in the next subchapter.

For now, your project looks like this:

```
kodai/
  Cargo.toml
  Cargo.lock
  src/
    main.rs
    lib.rs
```

This is the foundation everything builds on.

## Binary vs. Library Crates

Rust distinguishes between two kinds of crates:

| | Binary Crate | Library Crate |
|---|---|---|
| Entry file | `src/main.rs` | `src/lib.rs` |
| Output | An executable binary | Code other crates can import |
| Has `main()`? | Yes, required | No |
| Can have both? | Yes — one project can have both | |

Your project has both: `src/main.rs` makes it a binary crate (you can run it), and `src/lib.rs` makes it also a library crate (its code can be imported). This is the standard pattern for CLI applications in Rust.

::: python Coming from Python
In Python, the distinction between a script and a library is informal — any `.py` file can be both. In Rust, `main.rs` and `lib.rs` formally define the binary and library sides of your project. The binary is a thin entry point that calls into the library, keeping your logic testable and reusable. Think of `main.rs` as your `__main__.py` and `lib.rs` as your package's `__init__.py`.
:::

## The tests/ Directory

Rust has two kinds of tests:

1. **Unit tests** — live inside source files, in `#[cfg(test)]` modules (you saw these in the Cargo Basics subchapter).
2. **Integration tests** — live in the `tests/` directory at the project root.

Integration tests import your library crate from the outside, just like an external user would:

```rust
// tests/integration_test.rs
use kodai::greet;

#[test]
fn test_greet_returns_welcome_message() {
    let result = greet("Alice");
    assert!(result.contains("Alice"));
    assert!(result.contains("Kodai"));
}
```

Create the `tests/` directory and this file, then run:

```bash
cargo test
```

Cargo finds and runs both unit tests (in `src/`) and integration tests (in `tests/`) automatically. No test discovery configuration needed.

## Workspaces (A Preview)

As projects grow very large, Rust supports *workspaces* — multiple crates in a single repository that share dependencies and compile together. You will not need workspaces for this book, but it is worth knowing they exist. A workspace looks like:

```
my-workspace/
  Cargo.toml          # Workspace root
  kodai-cli/           # Binary crate
    Cargo.toml
    src/main.rs
  kodai-core/          # Library crate
    Cargo.toml
    src/lib.rs
```

Production coding agents often use this pattern to separate the CLI frontend from the core logic. For now, a single crate with `main.rs` and `lib.rs` gives you everything you need.

## Key Takeaways

- Rust projects have a conventional layout: `Cargo.toml` at the root, source code in `src/`, build artifacts in `target/`, and integration tests in `tests/`.
- Split your project into `src/main.rs` (thin binary entry point) and `src/lib.rs` (application logic) early — this keeps code testable and organized.
- `Cargo.lock` pins exact dependency versions. Commit it for binary projects to guarantee reproducible builds.
- The `target/` directory contains all compiled artifacts. Never commit it, and use `cargo clean` to reclaim disk space when needed.
- Plan your module structure early. The directory layout you choose now scales to the full coding agent you build throughout this book.
