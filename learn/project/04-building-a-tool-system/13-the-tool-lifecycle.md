# The Tool Lifecycle

> **What you'll learn:**
> - How to trace the complete journey of a tool call through all system components end to end
> - How the tool lifecycle integrates with the agentic loop's state machine from Chapter 3
> - How to identify bottlenecks and failure points in the lifecycle for debugging and optimization

You have now built every piece of the tool system: the trait, the schemas, the registry, the dispatch, the validation, the execution wrapper, the error handling, and the observation formatting. Each subchapter focused on one component. In this subchapter you step back and trace a single tool call through the entire system, from the moment the model decides to use a tool to the moment the observation feeds back into the next LLM call. Understanding this end-to-end flow is essential for debugging, performance tuning, and extending the system.

## The Seven Stages

A tool call passes through seven stages. Let's trace a concrete example: the model wants to read `src/main.rs`.

### Stage 1: Tool Declaration

Before the model can call a tool, it must know the tool exists. When you construct the API request, you include the `tools` array generated by `registry.tool_definitions()`:

```json
{
  "model": "claude-sonnet-4-20250514",
  "max_tokens": 4096,
  "system": "You are a helpful coding agent...",
  "tools": [
    {
      "name": "read_file",
      "description": "Read the contents of a file at the given path...",
      "input_schema": {
        "type": "object",
        "properties": {
          "path": { "type": "string", "description": "The file path to read." }
        },
        "required": ["path"]
      }
    }
  ],
  "messages": [
    { "role": "user", "content": "Read src/main.rs and explain what it does." }
  ]
}
```

The model reads the tool definitions and the user's request. It decides that `read_file` is the right tool and constructs the arguments based on the schema.

### Stage 2: Tool Call Emission

The model produces a response with `stop_reason: "tool_use"` and a `tool_use` content block:

```json
{
  "role": "assistant",
  "content": [
    {
      "type": "text",
      "text": "I'll read the file to understand the code."
    },
    {
      "type": "tool_use",
      "id": "toolu_01XFDUDYJgAACzvnptvVer8z",
      "name": "read_file",
      "input": { "path": "src/main.rs" }
    }
  ],
  "stop_reason": "tool_use"
}
```

Notice that the model can include text alongside the tool call. This text is the model "thinking out loud" -- explaining why it is calling the tool. Your agentic loop detects the `tool_use` stop reason and extracts the tool call.

### Stage 3: Registry Lookup

Your dispatch function receives the `ToolUse` struct and calls `registry.get("read_file")`. The registry looks up the name in its `HashMap` and returns a reference to the `ReadFileTool` implementation.

If the tool is not found (the model hallucinated a tool name), the dispatch function returns an error `ToolResult` immediately, skipping stages 4-5.

```rust
let tool = match registry.get(&tool_use.name) {
    Some(t) => t,
    None => return ToolResult {
        tool_use_id: tool_use.id.clone(),
        content: format!("Unknown tool: {}", tool_use.name),
        is_error: true,
    },
};
```

### Stage 4: Input Validation

The validation layer checks the model's input against the tool's JSON Schema:

```rust
// Does {"path": "src/main.rs"} match the schema?
// - "path" is a string: yes
// - "path" is in required: yes, and it is present
// Result: valid
```

If validation fails, the dispatch function returns an error `ToolResult` with the validation messages. The tool's `execute` method is never called.

### Stage 5: Tool Execution

The execution wrapper calls `tool.execute(&input)` with panic recovery and timing:

```rust
// Inside ReadFileTool::execute:
// 1. Extract "path" from input -> "src/main.rs"
// 2. Call std::fs::read_to_string("src/main.rs")
// 3. Format with line numbers
// 4. Return Ok("Contents of src/main.rs:\n   1 | fn main() {\n   2 | ...")
```

The execution wrapper measures the duration and catches any panics. If the tool takes too long or panics, a `ToolError::SystemError` is returned instead of the tool's normal output.

### Stage 6: Result Formatting

The dispatch function wraps the execution result in a `ToolResult`:

```rust
ToolResult {
    tool_use_id: "toolu_01XFDUDYJgAACzvnptvVer8z",
    content: "Contents of src/main.rs:\n   1 | fn main() {\n   2 |     ...",
    is_error: false,
}
```

If the output is too large, it gets truncated. The `tool_use_id` is preserved from the original request.

### Stage 7: Observation Feeding

The `ToolResult` is converted into a `tool_result` content block and appended to the conversation as a user message:

```json
{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "toolu_01XFDUDYJgAACzvnptvVer8z",
      "content": "Contents of src/main.rs:\n   1 | fn main() {\n   2 | ..."
    }
  ]
}
```

The conversation history now contains the original user request, the assistant's tool call, and the tool result. The loop sends this updated conversation back to the API for the next turn.

## The Lifecycle in Code

Here is how the seven stages map to your Rust code. This is the dispatch pipeline assembled from all the previous subchapters:

```rust
use serde_json::{json, Value};
use std::collections::HashMap;
use std::fmt;
use std::time::Instant;

#[derive(Debug)]
enum ToolError {
    InvalidInput(String),
    ExecutionFailed(String),
    SystemError(String),
}

impl fmt::Display for ToolError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            ToolError::InvalidInput(msg) => write!(f, "Invalid input: {}", msg),
            ToolError::ExecutionFailed(msg) => write!(f, "Execution failed: {}", msg),
            ToolError::SystemError(msg) => write!(f, "System error: {}", msg),
        }
    }
}

trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn input_schema(&self) -> Value;
    fn execute(&self, input: &Value) -> Result<String, ToolError>;
}

struct ToolUse { id: String, name: String, input: Value }
struct ToolResult { tool_use_id: String, content: String, is_error: bool }

struct ToolRegistry { tools: HashMap<String, Box<dyn Tool>> }

impl ToolRegistry {
    fn new() -> Self { ToolRegistry { tools: HashMap::new() } }
    fn register(&mut self, tool: Box<dyn Tool>) {
        let name = tool.name().to_string();
        self.tools.insert(name, tool);
    }
    fn get(&self, name: &str) -> Option<&dyn Tool> {
        self.tools.get(name).map(|t| t.as_ref())
    }
    fn tool_definitions(&self) -> Vec<Value> {
        self.tools.values().map(|t| json!({
            "name": t.name(),
            "description": t.description(),
            "input_schema": t.input_schema()
        })).collect()
    }
}

const MAX_OUTPUT_CHARS: usize = 50_000;

/// The complete tool dispatch pipeline: lookup -> validate -> execute -> format.
fn dispatch_tool_call(registry: &ToolRegistry, tool_use: &ToolUse) -> ToolResult {
    let start = Instant::now();

    // Stage 3: Registry lookup
    let tool = match registry.get(&tool_use.name) {
        Some(t) => t,
        None => {
            eprintln!("[dispatch] Unknown tool: {} ({:?})", tool_use.name, start.elapsed());
            return ToolResult {
                tool_use_id: tool_use.id.clone(),
                content: format!("Unknown tool: {}", tool_use.name),
                is_error: true,
            };
        }
    };

    // Stage 4: Input validation (simplified here)
    // In production, validate against tool.input_schema() with jsonschema crate

    // Stage 5: Execution with panic recovery
    let exec_result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        tool.execute(&tool_use.input)
    }));

    let duration = start.elapsed();
    eprintln!("[dispatch] {} completed in {:?}", tool_use.name, duration);

    // Stage 6: Result formatting
    match exec_result {
        Ok(Ok(output)) => {
            let content = if output.len() > MAX_OUTPUT_CHARS {
                format!(
                    "{}\n\n[Output truncated: {} of {} chars shown]",
                    &output[..MAX_OUTPUT_CHARS], MAX_OUTPUT_CHARS, output.len()
                )
            } else {
                output
            };
            ToolResult {
                tool_use_id: tool_use.id.clone(),
                content,
                is_error: false,
            }
        }
        Ok(Err(e)) => ToolResult {
            tool_use_id: tool_use.id.clone(),
            content: e.to_string(),
            is_error: true,
        },
        Err(_panic) => ToolResult {
            tool_use_id: tool_use.id.clone(),
            content: format!("System error: tool '{}' panicked", tool_use.name),
            is_error: true,
        },
    }
}

/// Stage 7: Convert ToolResult to API message format.
fn tool_result_to_message(result: &ToolResult) -> Value {
    let mut block = json!({
        "type": "tool_result",
        "tool_use_id": result.tool_use_id,
        "content": result.content,
    });
    if result.is_error {
        block["is_error"] = json!(true);
    }
    json!({ "role": "user", "content": [block] })
}
```

The `eprintln!` calls provide basic observability. In production, you would use a structured logging library like `tracing`, but `eprintln!` is sufficient for development.

## Failure Points

Each stage in the lifecycle has a characteristic failure mode:

| Stage | Failure | Recovery |
|-------|---------|----------|
| 1. Declaration | Tool schema is malformed | API returns error; fix schema |
| 2. Emission | Model hallucinates arguments | Validation catches it (Stage 4) |
| 3. Lookup | Model uses wrong tool name | Error result lists available tools |
| 4. Validation | Arguments violate schema | Error result with specific issues |
| 5. Execution | Tool fails (file not found, etc.) | Error result with actionable message |
| 6. Formatting | Output too large | Truncation with notice |
| 7. Feeding | ID mismatch | API error; this is a bug in your code |

The most dangerous failure is Stage 7: an ID mismatch. This is always a bug in your dispatch code (you passed the wrong `tool_use_id`), not a model error. Your tests should verify that IDs are preserved (as you wrote in subchapter 12).

::: python Coming from Python
The lifecycle stages are the same in Python-based agents. The difference is in how failures manifest. In Python, a tool execution failure raises an exception that you catch. In Rust, it returns a `Result` that you match on. The Rust approach makes it harder to accidentally swallow an error -- the compiler requires you to handle every `Result`. In Python, a missing `except` clause means an unhandled exception propagates and potentially crashes the loop.
:::

## Debugging the Lifecycle

When something goes wrong, you need to identify *which stage* failed. Add logging at each stage boundary:

```rust
fn dispatch_with_logging(
    registry: &ToolRegistry,
    tool_use: &ToolUse,
) -> ToolResult {
    eprintln!("[tool:{}] Dispatching (id: {})", tool_use.name, tool_use.id);

    let tool = match registry.get(&tool_use.name) {
        Some(t) => {
            eprintln!("[tool:{}] Found in registry", tool_use.name);
            t
        }
        None => {
            eprintln!("[tool:{}] NOT FOUND in registry", tool_use.name);
            return ToolResult {
                tool_use_id: tool_use.id.clone(),
                content: format!("Unknown tool: {}", tool_use.name),
                is_error: true,
            };
        }
    };

    eprintln!("[tool:{}] Input: {}", tool_use.name, tool_use.input);

    let start = Instant::now();
    let result = tool.execute(&tool_use.input);
    let duration = start.elapsed();

    match &result {
        Ok(output) => {
            eprintln!(
                "[tool:{}] Success in {:?} ({} bytes output)",
                tool_use.name, duration, output.len()
            );
        }
        Err(e) => {
            eprintln!("[tool:{}] Error in {:?}: {}", tool_use.name, duration, e);
        }
    }

    match result {
        Ok(output) => ToolResult {
            tool_use_id: tool_use.id.clone(),
            content: output,
            is_error: false,
        },
        Err(e) => ToolResult {
            tool_use_id: tool_use.id.clone(),
            content: e.to_string(),
            is_error: true,
        },
    }
}
```

This gives you a trace like:

```
[tool:read_file] Dispatching (id: toolu_01XFD)
[tool:read_file] Found in registry
[tool:read_file] Input: {"path":"src/main.rs"}
[tool:read_file] Success in 1.2ms (342 bytes output)
```

Or for a failure:

```
[tool:read_file] Dispatching (id: toolu_02ABC)
[tool:read_file] Found in registry
[tool:read_file] Input: {"path":"nonexistent.rs"}
[tool:read_file] Error in 0.8ms: Execution failed: Cannot read 'nonexistent.rs': No such file or directory
```

This trace tells you immediately which stage failed and how long each stage took.

::: wild In the Wild
Claude Code logs the full tool lifecycle for debugging: the tool name, the input (sanitized for sensitive content), the execution duration, and the result size. It also tracks tool call counts per conversation to detect potential infinite loops (where the model keeps calling the same tool with the same arguments). OpenCode provides similar telemetry through its structured logging system.
:::

## Key Takeaways

- A tool call passes through seven stages: declaration, emission, lookup, validation, execution, formatting, and feeding. Each stage has a characteristic failure mode.
- The lifecycle integrates with the agentic loop at two points: tool definitions go into the API request (Stage 1), and tool results come back as observations (Stage 7).
- ID mismatch (Stage 7) is the most dangerous failure because it is always a bug in your code and causes API errors. Test for ID preservation.
- Add logging at each stage boundary to create a trace that shows exactly where a failure occurred and how long each stage took.
- Understanding the full lifecycle is essential for debugging: when the agent behaves unexpectedly, trace the tool call through all seven stages to find the source.
